# ADR-001: Core Architecture — CLI-First, SQLite, Offline-First

**Status:** Accepted  
**Date:** 2025-03-02  
**TODO:** Sprint 0, Section 1 — Architecture decision record

---

## Context

Phaicull is a photo culling assistant that must work offline, protect user privacy, and support multiple native UIs (macOS SwiftUI, Windows WinUI). We need a clear architecture that defines how the Python core, database, and UIs interact.

---

## Decision

### 1. CLI-First Execution Model

The Python core is a **standalone CLI tool**. It is the single source of truth for scanning, analysis, and persistence. UIs do not embed or import the core; they invoke it as a subprocess or read its outputs.

- **Entry point:** `core/cli.py` (Typer app)
- **Invocation:** `phaicull scan <path>`, `phaicull report`, etc.
- **Implication:** The CLI must be fully functional without any UI. All logic lives in the core.

### 2. SQLite Default DB

Each photo directory (project) has **one SQLite database** at `{project}/phaicull/phaicull.db`. No server, no remote DB.

- **Engine:** raw `sqlite3` (see [DEC-001](decisions.md#dec-001-sqlite-access--raw-sqlite3-vs-sqlalchemy))
- **WAL mode:** Enabled on every connection for better concurrency
- **Schema versioning:** `schema_version` table + custom migration script (see [DEC-004](decisions.md#dec-004-migration-strategy--custom-script-vs-alembic))

### 3. Offline-First, Plugin-Based AI

- **No cloud:** All processing runs locally. No telemetry, no runtime model downloads.
- **Models:** Stored in `.models/` (git-ignored). Bundled with the app or installed separately.
- **Plugin-based:** AI models (CLIP, aesthetic scorer) are swappable. Users can add personal models without changing core code.
- **Privacy:** Family photos never leave the device.

### 4. UI Integration Contracts

UIs (macOS, Windows) integrate via well-defined contracts. These are **stable APIs** — changes require versioning and migration notes.

| Contract | Description | Format |
|----------|-------------|--------|
| **JSON output** | CLI stdout for machine consumption | JSON object/array; field names and types are stable |
| **Progress reporting** | Real-time scan progress for UI progress bars | Structured log stream (loguru JSON); schema: `event`, `percent`, `current`, `message` |
| **DB schema** | UI reads SQLite directly for grid view, filters | Project DB: `files`, `metrics`, `groups`. Registry DB: `projects`. Column names and types are stable. |
| **Exit codes** | CLI success/failure | `0` = success, non-zero = failure (POSIX) |

**UI integration modes:**
1. **Subprocess:** UI spawns `phaicull scan ...`, parses stdout/stderr for JSON and progress
2. **Direct read:** UI opens `phaicull.db` and queries tables (e.g., for grid view while scan runs)

---

## Consequences

- **Positive:** Clear separation of concerns; UIs can be rebuilt without touching the core; offline and privacy-preserving by design
- **Negative:** UI must handle subprocess lifecycle and parsing; DB schema changes require coordination
- **Neutral:** No JSON-RPC or similar in MVP; may revisit in later sprints if needed
